# Wattly — 電力SaaS（構想中）

## プロジェクト概要

電力業界向けSaaS。まずはアフィリエイトメディアとしてスタートし、ドメイン知識とドメインパワーを蓄積してからSaaS事業の方向性を決定する。

### 戦略
1. **Phase 0**: 電力アフィリエイトメディア運営でドメイン知識 + ドメインパワー + 収益を獲得
2. **Phase 1**: 蓄積した知識をもとにSaaSの方向性を決定（月3〜5万PV到達が目安）
3. **Phase 2**: MVP開発（メディアの読者が見込み客になる）

**Casinohubとの違い**: Casinohubは「作るべきSaaSが明確（チップ管理）」なので即開発。Wattlyは「何のSaaSを作るかが未定」なのでメディア運営で知見を貯めてから判断する。ただしメディア運営中に明確なペインが見つかったらPVに関係なく即動く。

## プロジェクト構成

```
wattly/
├── hp/               ← メディアサイト（Next.js）→ アフィリエイト + SEO
├── marketing/        ← 記事原稿
├── 事業運営/          ← 事業運営タスク管理
└── images/           ← ロゴ・素材
```

### URL設計
- 全ての記事を `/column/` 配下に統一する
- 例: `wattly.jp/column/electricity-bill-structure`
- 消費者向け・事業者向けの区別はURLではなくタグやカテゴリで管理

## 共通ルール

- **言語**: 日本語で作業・出力
- **金額**: 全て税込で記載し、表示時は「（税込）」を必ず付記する
- **日付**: YYYY-MM-DD 形式
- **コミットメッセージ**: 日本語、簡潔に
- **秘密情報**: `.env`, credentials, API keyは絶対にコミットしない

### ワークフロー
- **確認不要**: wattly配下のプロジェクトではコマンド実行・push 含め一切確認を求めず自律的に進める
- **「リリースして」= push まで含む**: コミット → push まで一気にやる
- **記事リリース**: draft から `hp/content/column/{date}/` に MDX 変換して配置 → リリース後draftファイルは削除 → コミット → push（contentが唯一の正）

### 文体・トーン（記事・メディア）
- 同じ目線で語るトーンを使う
- 上から目線・偉そうな口調はNG
- 断言できないことは断言しない
- 競合の名指し・比較・批判はしない
- 電気に詳しくない一般消費者にもわかりやすく

### アフィリエイト導線設計
- ASP未登録のため今はリンクを貼らないが、**将来差し替えやすい構造**にしておく
- 具体的なサービス名を出すか「複数社を比較検討するのがおすすめ」の一文を自然に入れる
- CTA（比較表・シミュレーション等）を記事内に設置する想定で構成を組む

### 記事執筆ルール

#### 文字数
- 3,000〜7,000字（検索意図に応じて柔軟に）
- 情報系記事: 長め / ハウツー・実用系: 短くてもOK

#### フォーマット要件
- **タイトル（H1）**: キーワードを前半に配置、最大32文字
- **見出し階層**: H1 → H2 → H3 を厳守（飛ばさない）
- **リード文**: 読者の課題に共感 → 記事で何がわかるか説明（150〜200字）
- **メタディスクリプション**: 最大120文字
- **スラッグ**: 英語ハイフン区切り（例: `electricity-bill-structure`）
- **内部リンク**: 同クラスター内2〜3本、クラスター横断もOK（自然な文脈で）
- **画像alt**: キーワードを含める

#### CTA設計
- 記事末尾（まとめセクションの下）にCTAを設置
- 現時点はアフィリエイトリンクなしだが、将来差し替え可能な構造にする
- 読者の行動を促す具体的な一文を入れる（「まずは○○から始めてみてください」等）

### 記事作成ワークフロー（3ステップ）

記事を書く前に必ずこの3ステップを踏む:

**Step 1: 角度ブレスト（必須・執筆前にユーザー確認）**
- テーマに対して構造・切り口・視点を3〜5案出す（読者が「え？」と思うもの）
- 常識・正論・ありきたりな答えは除外
- 技法: 問いの逆転 / 他業界からの輸入 / 具体シーンから入る / データファースト / 視点の転換 / 正解を言わない
- **ユーザーが選んだ案が確定するまで執筆しない**
- **draft作成前（ユーザーからテーマを受け取った段階）でも必ずStep 1を先に提示する**

**Step 2: 具体 ↔ 抽象の設計**
- **具体（入口）**: 読者が自分の経験から「あるある」と思える場面描写
- **抽象（本質）**: より深い構造・問いで「なるほど」と思わせる
- 具体 → 抽象の順で書く（逆にしない）

**Step 3: 執筆**
- Step 1-2で方向が決まってから書き始める
- 毎回同じ構成（問題提起→なぜダメか→代替案→まとめ）にしない
- 結論が構成から予測できない記事にする

### ライティングテクニック
- **断言する**: 「〜と言われています」「〜かもしれません」を減らす。根拠があることは言い切る
- **具体シーンを描く**: 「真夏の14時、エアコンをつけた瞬間にブレーカーが落ちた」のような場面
- **書き手の意見を入れる**: 「正直、これはもったいない」「ここが一番見落とされがちだ」
- **リズムをつくる**: 長文＋短文、ダッシュ、問いかけ、体言止めを織り交ぜる
- **読者に直接語りかける**: 「あなたの家の電気代、先月いくらだったか覚えていますか？」
- **同テーマの表現変化（必須）**: 複数記事で同じトピックに触れる場合、毎回フォーマットを変える（箇条書き↔散文、俯瞰↔具体シーン、三人称↔一人称等）。読者が「前にも同じ文章を読んだ」と感じたらNG

### 禁止コンテンツ
- 虚偽・誇張した表現
- 競合サービスの名指し比較・批判（エネチェンジ等の名前を出して「こっちが上」は絶対NG）
- 法的に不正確な記述（電力自由化制度・再エネ賦課金等は正確に）
- 他サイトのコピー・リライト
- 「いかがでしたか」等の定型フレーズ
- 上から目線の「教えてあげる」口調

### セルフレビュー工程（必須）

全ての成果物を提出する前に、「ユーザーがさらに優秀になったバージョン」を模倣して一度レビュー・ブラッシュアップを行ってから提出すること。

#### セルフレビューチェックリスト（記事）
- [ ] タイトル32文字以内、キーワードが前半にあるか
- [ ] H1→H2→H3の階層が守られているか
- [ ] リード文で読者の課題に共感し、記事の内容を予告しているか
- [ ] メタディスクリプションが120文字以内か
- [ ] 断言できないことを断言していないか
- [ ] 具体シーンが最低1つあるか
- [ ] 同テーマの既存記事と表現が被っていないか
- [ ] 内部リンクが適切に設置されているか
- [ ] CTAが記事末尾にあるか
- [ ] アフィリエイト導線が将来差し替え可能な構造になっているか
- [ ] 数値データの出典が確認済みか

### Ralph Loop開発フロー（PRD駆動の自律開発）

1. **PRD作成**: `PRD.md` にタスク一覧を作成
2. **PRD深掘り**: 各タスクを実装・テスト・検証・エッジケースまで含めたサブタスクに分解
3. **Ralph Loop実行**: PRDのタスクを上から順に消化。全タスクが `[x]` + 検証PASSまで止まらない
4. **完了タスクは即座に `[x]` 更新** → コミット → 次へ

長時間作業中は、ユーザーが非同期で確認できるように以下のフォーマットで出力する:


### SEO戦略（2層構造）

エネチェンジ・価格.comが「比較・おすすめ」系を押さえているため、正面衝突は避ける。**視座系（看板記事）と実用系の2層構造**で攻める。

**視座系（看板記事）: 月1〜2本、しっかり書く**
- 「問いを立てる」「前提を疑う」「構造を俯瞰する」切り口
- 例: 「電気代は本当に高いのか？」「電力自由化は誰のためだったのか」「なぜ電気料金はわかりにくいのか」
- 競合が書かない領域。AIでも量産できない（視座が必要）
- SNSシェア・被リンクでドメインパワーを育てる（検索流入ではなくSNS起点）
- 実用記事への内部リンクのハブになる → 実用記事のSEO評価を底上げ
- SaaS展開時の思想的土台にもなる（問題提起→だからツールを作った、の流れ）

**実用系: 効率的に数を揃える**
- 節約・比較・制度解説などの実用記事
- 視座系で育てたドメインパワー+内部リンクの恩恵でSEO上位を狙う
- アフィリエイト成約はこちらで担う
- ロングテール（個別電力会社レビュー等）と法人特化（高単価×低競争）を優先

**収益の流れ:**
```
視座系記事（SNS・被リンク） → ドメインパワー向上
  → 実用記事のSEO順位UP → アフィリエイト成約
視座系記事 → 内部リンク → 実用記事への回遊 → 成約
```
